# Phase 260321-147 Plan 01: Multi-Account Support Summary

## One-Liner

Multi-account support with named accounts, secure keychain storage, account switching via menu bar dropdown, and automatic migration from legacy single-key setup.

## What Was Built

### Account Model (`Account.swift`)
- `Account` struct with `id: UUID`, `name: String`, `apiKey: String`
- Conforms to `Codable`, `Identifiable`, `Hashable`
- `obfuscatedKey` computed property for safe display (shows only last 4 characters)

### KeychainService Extensions
- Multi-account storage using JSON-encoded array
- `saveAccount(_:)` - creates or updates account
- `loadAccounts()` - retrieves all accounts with auto-migration
- `deleteAccount(id:)` - removes account by ID
- `saveActiveAccountId(_:)` / `loadActiveAccountId()` - active account tracking
- Migration: automatically creates "Default" account from legacy single key

### UsageViewModel Account Management
- `@Published var accounts: [Account]` - all configured accounts
- `@Published var activeAccount: Account?` - currently active account
- Account management methods: `addAccount`, `removeAccount`, `switchToAccount`, `updateAccount`
- Updated `refresh()` to use `activeAccount?.apiKey`
- Updated `hasAPIKey` to check for active account
- Legacy `saveAPIKey` method preserved for backward compatibility

### SettingsView UI
- Account list section replacing single API key field
- Account rows showing name + obfuscated key with "Active" badge
- Add account sheet with name and API key fields
- Edit account functionality (name always, key optional)
- Swipe-to-delete support
- Empty state placeholder ("No Accounts Added")

### MenuBarView UI
- Account picker dropdown when multiple accounts exist
- Single account shown as subtle label when only one
- Updated empty state text from "API Key Required" to "Add an Account"

## Deviations from Plan

None - plan executed exactly as written.
